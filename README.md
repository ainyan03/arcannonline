# ArcannonLine — 魔法弾幕オンライン共闘（試作）

> arcane (魔法) + cannon (砲) + online

オープンフィールドを魔法使いたちが飛び回り、魔法弾幕で中央の「魔力灯」を守る
常時参加型・協力拠点防衛ブラウザゲームの試作。
**自前サーバを一切持たず**、GitHub Pages の静的配信だけで動作する。

## アーキテクチャ

ネットワーク層はすべて自前実装（外部P2Pライブラリ不使用）:

```
GitHub Pages（静的配信のみ）
    │ 配信
    ▼
ブラウザA ── 公開Nostrリレー(wss) ── ブラウザB   … ブートストラップ (nostr.ts)
    │        プレゼンス通知と SDP 交換 (署名付き ephemeral イベント)
    │
    └────── WebRTC DataChannel メッシュ (peer.ts / mesh.ts) ──── ブラウザB
             ├ state:    プレイヤー位置 (~15Hz)・担当NPC群 (5Hz、非信頼・非順序)
             ├ reliable: PEX・チャット・発射/弾消滅・拠点ダメージイベント
             └ PEX: 既知ピア一覧の交換 + シグナリングのメッシュ中継
                    → リレー依存は事実上「最初の1本の接続確立」のみ
```

- **ピアID = secp256k1 公開鍵**。Nostr イベントは BIP340 schnorr 署名付きで、
  正規化本文からイベントIDを再計算したうえで署名を検証する
- P2Pで受信するstate・PEX・シグナル・発射・チャットは、型だけでなく
  実行時にも文字数・配列数・数値範囲・公開鍵形式を検証する
- 0.1秒間隔の通常弾は、対応ピア同士では最大250ms・8イベントまでReliable
  メッセージへまとめる。旧版・版数不明ピアへは従来の個別送信へ自動フォールバックする
- シグナリング封筒には送信元ピア鍵の schnorr 署名を付け、経路自体に署名がない
  メッシュ中継分は封筒署名を検証してから扱う（中継者による送信元なりすましを排除。
  リレー経由はNostrイベント署名で担保）
- **バニラICE**: ICE候補はSDPに同梱して1発で送る（トリクル不使用）。
  リレーへのイベント数を1接続あたり2件に抑え、レート制限を回避する
- **オファーはID小さい側のみ発行**（perfect negotiation + glare回避）
- 演算は 2D 平面 (x, y)。描画時に XZ 平面へ写像し Three.js で 3D 俯瞰表示
- NAT 越えは STUN のみ（TURN なし）。直接通信が確立できない相手とは同期しない割り切り
- 被弾判定は自己申告制（チート対策はスコープ外）
- 想定スケール: 同時 16〜32 人、平均 24 人

### アカウントと GitHub 連携（任意）

- 秘密鍵を localStorage に永続化し、公開鍵 (= ピアID) を継続的なアカウントとして扱う。
  頭上ラベルにはピアID由来の identicon を表示する（同一ブラウザの複数タブは
  Web Locks で先着1タブだけが永続鍵を使い、以降のタブは使い捨て鍵で参加）
- 参加画面の「GitHub 連携」で Firebase Authentication (GitHub OAuth) にログインすると、
  identicon が GitHub アバター + 認証バッジ（緑チェック）に変わる。未ログインでも
  プレイに制限はない（Firebase SDK は分離チャンクで、ログイン利用時のみロード）
- 連携中は GitHub の @ユーザー名で参加する（名前欄はロック、解除で手入力名に戻る）。
  @ユーザー名は ID トークンに含まれないため、ログイン時に Firebase の displayName へ
  ピア公開鍵と一緒に書き込み、Firebase署名対象の name クレームとして束縛する。
  頭上ラベルはフル表示する（位置同期 state 上の名前は互換性のため16文字まで）
- 認証情報の配布にサーバは使わない: 自分の Firebase ID トークン (JWT) を reliable
  チャネルでピアへ配り、受信側が Google の公開鍵 (JWKS) を使って署名・発行元・
  GitHubプロバイダID・有効期限・ピア公開鍵束縛を独立検証してから表示に反映する。
  別ピアによるトークン転載は拒否し、期限切れ・連携解除時はidenticonへ戻す
- 認証バッジが保証するのは「GitHubログインと現在のピア鍵の束縛」。表示名自体は
  Firebaseプロフィールへ本人が設定する自己申告値である

### 耐障害設計（実地テストからのフィードバック）

- 状態送信は rAF でなく setInterval + イベント駆動（バックグラウンドタブの
  スロットリング対策。接続確立時の即時送信と受信応答型 keepalive を併用）
- `pagehide` で退室通知（スマホの画面ロック対策）、復帰時は自動再参加
- 10秒間無通信のリモートはゴーストとみなして表示から除去
- 被弾判定は rAF と state 送信タイマーの両方で駆動する（rAF が止まる
  バックグラウンドタブが「攻撃の効かない置物」になるのを防ぐ）
- 5分間入力がないクライアントは AFK として自動退室し参加画面へ戻る
  （放置タブのキャラ残留対策。bot モードは対象外）
- リレーのイベントリプレイは `created_at` の鮮度チェックで排除
- HUD に relays / peers / ice / tx / rx の診断表示（rtc-debug.ts が
  ICE 状態遷移と採用経路をコンソールへ記録）
- プレゼンスと PEX にプロトコル版数 (`PROTO_VERSION`) を同梱。自分より新しい
  版数のピアを見つけたクライアントは更新バナーを表示する（接続・プレイは継続。
  互換性が壊れる変更を入れる際に版数を +1 する）
- state/reliable両チャネルの開通を接続成立条件とし、open待ちキューを件数・総量で制限する
- PEX 1件からの新規接続試行と全体の同時ネゴシエーション数を制限し、偽IDによる枠占有を抑える

## 開発

```bash
npm install
npm run dev          # http://localhost:5173  (localhost はセキュアコンテキスト)
npm run dev:https    # https://<IP>:5174     (LAN内の他端末から。自己署名証明書)
npm test             # parser/VM/通信検証/決定論シミュレーションの単体テスト
npm run check        # typecheck + test + production build
```

ブラウザ（タブ）を2つ以上開いて参加すると、互いのキャラクターが見える。
ブートストラップに公開 Nostr リレーを使うため、開発時もインターネット接続が必要。
開発ビルドのクライアントはプロトコル版数を申告しない（未公開の版数が
本番プレイヤーに「更新できない更新バナー」を出させるのを防ぐ）。
通常クライアント1つにつき小型敵が8体出現するため、1タブだけでも
拠点への進軍・攻撃・撃破・再出現を試せる。複数タブでは各タブの担当分が加算される。

## デプロイ (GitHub Pages)

`main` ブランチへの push で `.github/workflows/deploy.yml` が起動し、
Vite ビルド成果物 (`client/dist`) が GitHub Pages に公開される。
リポジトリ設定で Pages のソースを「GitHub Actions」にしておくこと。

## 操作方法

タッチ (スマホ):
- 1本指: 押した地点へ移動。押したままドラッグ/フリックで移動先が指に追従
- 2本指 (地図アプリ風): ピンチ=ズーム、ひねり=フィールド回転、上下=見下ろし角度
- 左下の仮想スティック: 全方向移動（発射ボタンと同時操作向け）
- 右下の発射ボタン: ボム発射（単押しで1回、長押しで連射。上のチップは
  自分のクラスのボム名）
- 画面最下部の入力枠: チャット

マウス / キーボード (PC):
- 左ドラッグ: 押した地点へ移動 (追従)、右ドラッグ: 視点回転、ホイール: ズーム
- WASD / 矢印キー: 移動（カメラ基準）
- Space: ボム発射
- Enter: チャット入力へ
- 右上の 🔊 ボタン: 効果音のオン/オフ（設定はブラウザに保存）

## ゲームルール（試作段階）

- **協力プレイ (PvE)**: プレイヤー同士は協力して敵と戦う。フレンドリーファイアはなく、
  ダメージを受けるのは敵の弾のみ（プレイヤーの弾は他プレイヤーをすり抜ける）
- **岩と遮蔽**: フィールドには円形の立ち岩があり、移動を遮り、弾も岩に当たると
  消える（敵味方とも。岩を盾にした撃ち合いができる）。魔力灯も実体があり
  重なれない。配置は全クライアント共通の静的定数で、衝突は各自ローカルに
  解決するため追加の同期はない（弾道は決定論再現なので岩ヒットも全員一致する）
- **拠点防衛**: 敵はフィールド中央の魔力灯へ進軍して弾幕で攻撃する。
  魔力灯はHP 100で、直近30秒のダメージが時間経過で回復する。HP 0で消灯し、
  ゲージが全快すると再点火する（0付近での消灯/再点火のちらつきを防ぐ）。
  消灯中の魔力灯を敵は狙わず、流れ弾もダメージにならない（回復は必ず進む）。
  消灯中の敵は代わりにプレイヤーを狙って襲ってくる
- **反撃 (アグロ)**: 攻撃を受けて生き残った敵は、しばらく攻撃してきたプレイヤーを
  優先して狙う（点灯中でも有効。敵を撃って魔力灯から引き剥がす戦術が生まれる）
- 拠点命中は弾を所有するNPCの担当ピアが確定する。直近命中履歴を集合としてP2Pマージし、
  重複加算を防ぎつつ途中参加者にも現在の耐久状態を同期する
- HP 100。被弾すると弾の耐久度ぶんダメージ（被弾判定は自己申告制:
  自分のクライアントだけが自機の判定を行い、結果のHPを共有する）
- HP 0 でランダム地点にリスポーン（2秒無敵・点滅）。撃墜はチャットに通知
- 通常ショットは、射程内かつ自機の前方（左右90度未満）に生存中の敵がいる場合のみ
  自機の向きへ自動連射する。エネルギーは消費しない。**弾道はクラス別**で、
  さらに高速移動中はどのクラスもバラつきが増える（静止で収束する）:
  - 星: 0.1秒間隔の素直な連射（基準）
  - 箒: 0.08秒間隔の高連射・軽量弾（常に動いているぶん弾道は荒れやすい）
  - 月: 0.25秒間隔の重い狙撃弾。静止するとレーザーのように収束し、長射程
  - 花: 0.22秒間隔で3発同時の短射程ショットガン
- ボム発動時は短い無敵（0.8秒）がある
- 弾同士は敵味方が異なると衝突し、耐久度を削り合う（敵弾の撃ち落とし。
  プレイヤー弾同士・敵弾同士は干渉しない）
- 敵の弾は全プレイヤー共通の警告色（赤系）+ 大きめのハローで描画し、
  中心コアが白熱⇔警告色で明滅する（約4Hz、全敵弾同位相）。
  味方の弾（所有者のアバター色連動、自弾は淡黄）は安定発色で、一目で区別できる
- 弾の残存時間は省略時4秒・上限8秒（射程 ≒ 速度×残存時間。フィールドの
  端まで漂い続ける長射程弾を防ぐ）。消滅前の1秒間は縮小しながら消える
  （判定半径も一緒に縮み、見た目と当たりが食い違わない）
- プレイヤー弾は発射時の自機速度を50%引き継ぎ、移動方向へ弾幕全体が流れる
- チャットログはスクロールでき（最下部にいる間は自動追従）、途中参加時には
  在室中の各プレイヤーから**本人の直近発言だけ**（各最大30件）を受け取って
  会話の流れを復元する。他人の発言は転送せず、履歴の表示名も受信側が
  送信ピアの実体から解決するため、他人名義の発言の捏造はできない
  （代わりに、退室したプレイヤーの発言は引き継がれない）。
  発言IDで重複排除し、サーバなしでピア間の直接送信だけで完結する
- 発射・弾の衝突・被弾・撃破・参加/退出・チャット着信を効果音で通知する。
  音は外部ファイルなしでブラウザ内生成し、大量衝突時は発音頻度を制限する
- キャラクターは約二頭身のアニメ調低ポリゴン魔法使い。衣装プリセットで杖飛行・箒飛行を
  選び、色と魔女帽子/リボンを組み合わせられる
- **クラス別の移動特性**: 箒の魔女は高速・強慣性で**止まれない**（入力がなくても
  現在の向きへ飛び続け、岩は自動で避け、タップした目標地点は減速せず通過する。
  逆方向への入力で旋回はできるが最低速度は割らない）。月の魔導士はやや低速だが
  キビキビ動く。移動は自己申告のため同期への影響はない
- 各クライアントは小型敵8体を担当する。担当側だけがAI・HP・被弾を判定し、
  他クライアントはスナップショットを補間表示する。敵は中央の魔力灯へ進軍して
  3way弾幕を撃ち、HP 0から6秒後に別地点へ再出現する
- **ウェーブ制**: 敵は90秒周期の波で襲来する（襲来75秒 + 小休止15秒。
  小休止中は倒した敵が再出現しない）。第1〜5波と進むごとに敵の攻撃頻度と
  移動速度が上がり、第5波の後は第1波に戻る。スケジュールは壁時計から
  決定論的に算出する。接続ピアの時計ずれ中央値へ各自の時刻を寄せるため、
  サーバやリーダーなしでも大きな端末時計ずれの影響を抑える
- **敵は4種類**。波が進むほど特殊型が混ざる（種別は担当ピアが state で申告し、
  倒された敵は次の再出現からその波の編成に従う）:
  - **ウィスプ**: 標準の追跡型。3way弾（HP 24）
  - **インプ** (突進型): 高速で肉薄し、自爆して全方位に弾リングを撒く（HP 14）
  - **ガーゴイル** (砲台型): ほぼ居座り、遠距離から重く遅い狙い弾を撃つ（HP 40）
  - **ゴーレム** (重装型): 硬くて遅い盾。攻撃は控えめ（HP 64）
- **ボス「アークファントム」**: 第5波の後のボス段に外周から現れる（HP 600、
  90秒周期の1区分ぶん）。二重リング・狙い5way・回転スパイラルの弾幕を回す。
  ボス段の間は雑魚が再出現しない。撃破できなければ時間切れで退いていく。
  担当は接続中の最小ピアIDのクライアント（リーダー）で、リーダーが退室したら
  次のリーダーが最後に観測した HP を引き継いで担当し直す
- **エネルギー**: 最大200、回復15/秒。発射時にスクリプトの総コストを
  同じシードで空実行して算出し一括消費する（足りなければ発射不可）。
  1発あたり `0.5 + 0.5×(耐久-1) + 4×max(半径-0.2, 0)`。
  目安: 全方位リング≒29、双腕スパイラル≒26、前方ショット・狙い3way≒3。
  大量・高耐久・大玉ほど高コストになる。発射は長押しで連射でき、実行中の魔法が
  ある間は次を撃てない
- 安全弁としてオーナーごとの弾の同時存在数上限 300 発。超過時は発射を
  ブロックせず、**最も古い弾から寿命を前倒しして消す** (FIFO)。この規則は
  全クライアントが同一に適用するため追加の同期通信なしで一致する

## ボム（強攻撃）と弾幕エンジン

強攻撃はクラス（見た目プリセット）ごとに1種類固定の「ボム」で、切替UIは
持たない（星=全方位リング / 箒=前方ショット / 月=マジックミサイル /
花=双腕スパイラル）。プレイヤー弾は敵弾を相殺するため、高耐久のボムは
緊急時の弾消しとして機能する。
月のマジックミサイルだけは特殊で、発射時にボス優先で最大6標的（射程70）を
ロックし、1発ずつ時差をつけた曲線軌道のホーミング演出ののち必中でダメージが
入る（弾同士の相殺は経由しない。視覚エフェクトは各クライアント任意 = 同期不要、
ダメージの確定は標的NPCの担当ピアが行う）。飛行時間は距離に比例し
（見かけ速度一定、最長2.5秒）、標的がいない時は発射されない。
自由編集UIはゲーム導線を複雑にするため提供しない。内部の弾道定義には
最小DSLを使用する（[danmaku-scripts.ts](shared/src/danmaku-scripts.ts)）。
発射時は「スクリプトID・乱数シード・位置・向き・照準位置・時刻」を P2P 同期し、
各クライアントがシード付き乱数で決定論的に再現演算する（弾の座標は送らない）。
発射元と照準はイベント生成時点の値に固定し、stateの到着順が端末ごとに違っても
`wait`を含む多段弾幕が分岐しないようにしている。

- 実行系: 字句/構文解析 → generator ベースの VM（wait で中断・再開、1 tick/全体の
  命令数・生成弾数・60秒の実行時間上限つき）。tick は 60Hz 固定タイムステップで、バックグラウンドタブでも
  setInterval 経由で進行する
- 弾同士の当たり判定: 敵味方が異なる弾 (プレイヤー弾×敵弾) が接触すると、
  衝突時点の耐久度を互いに減算し合い、0 以下になった弾は消滅する
  （空間ハッシュで近傍のみ判定）。弾所有ピアIDから衝突確定担当を決定し、
  可換な耐久ダメージイベントを共有する。消滅・ダメージ通知が発射より先着しても保持する
- 受信側は伝送遅延ぶんを追いつき再生して発射側との位置ずれを抑える

## 分散NPC（試作）

全体管理者は置かず、NPC IDを `<担当peerId>:npc:<slot>` として各ピアが8体ずつ
所有する。所有ピアがAI、被弾、撃破、再出現を計算し、5Hzの状態配信と決定論的な
発射イベントを共有する。新規参加者には各ピアが自分のNPCだけを送る。
受信側はスナップショットを280ms遅延再生し、速度つき Hermite 補間 +
未着時の短時間外挿で 5Hz でも滑らかに表示する。

- 所有ピアが退出・無通信になると、そのピア配下のNPCだけが消える
- NPC同士の弾は相殺せず、プレイヤー弾とNPC弾は相殺する
- NPCの攻撃は既存の弾幕エンジンで再現され、担当ピアが拠点への命中を確定する
- 引き継ぎやリースは持たないため、ネットワーク分断後に合流してもNPC IDが競合しない
- 現段階ではドロップ、陣営、チーム得点、共通ボスは未実装

## 品質チェック

`main`以外へのpushとPull Requestでは `.github/workflows/ci.yml` が `npm run check` を実行する。
`main`へのpushでは同じチェックに成功した成果物だけをGitHub Pagesへデプロイする。

現在の主な制約:

- TURNを持たないため、STUNだけで直接接続できないネットワーク同士は対戦できない
- 被弾・HP・エネルギーは自己申告制で、競技向けのチート耐性は持たない
- ルームは単一フィールドで、32接続を超えるinterest managementは未実装

## スケール検証

- 単体ベンチ: コンソールで `__arcBench()` を実行すると、移動・場外カリング・
  弾同士の衝突判定に加え、静的な岩との移動区間衝突を含む tick の実測時間を
  弾数別に返す。
  実測 (Apple Silicon Mac / Chrome):
  - 衝突なし持続負荷: 8,000発 4.4ms / 20,000発 12.5ms / 50,000発 42ms per tick
  - 実戦相当 (4オーナー混在): 8,000発 2.5ms / 50,000発 8.3ms per tick
  - 岩判定込み (シナリオC): 8,000発 ≒ 6.0ms/tick と、衝突なし比 +1〜10% 程度
  - 現在の上限 (MAX_BULLETS=8192) では 60Hz の tick 予算に十分収まるため、
    **WASM 化は当面不要**。5万発を持続させる場合は SoA 化 / WASM が候補
- bot 20機 (プレイヤー20 + NPC 160) の再実測 (8体/クライアント化後):
  全対全メッシュが数分で確立して安定 (links 440/440)、bot あたり rx 約400msg/s。
  32接続上限までの前提は8体化後も変わらず
- 負荷試験: `?bot=1` を付けて開くとヘッドレス bot として参加する
  (自動徘徊・定期発射)。タブを複数開けば多人数接続を再現できる
